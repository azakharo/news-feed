# Roadmap: NewsFeed Virtualized Study Project

## Фаза 1: Бэкенд и Инфраструктура Данных

**Цель:** Создать фундамент системы: базу данных и API, поддерживающее детерминированную пагинацию (Cursor-based) и поиск.

* **Зависимости:** Установленные Docker, Node.js, NestJS CLI.
* **Вход:** Схема сущности `Post` с полями `content` и `aspectRatio`.
* **Выход/Артефакты:**
  * `docker-compose.yaml` (Postgres).
  * NestJS проект с настроенным TypeORM.
  * API эндпоинт `GET /posts`.
  * Seed-скрипт для генерации 10,000+ записей.
* **Definition of Done (DoD):**
  * [ ] База данных успешно инициализируется через Docker.
  * [ ] Seed-скрипт заполняет БД записями с разной длиной текста (от 10 до 2000 символов).
  * [ ] Запрос к API с параметром `cursor` возвращает строго следующие элементы (без дублей).
  * [ ] Параметр `search` корректно фильтрует записи через `ILIKE`.

## Фаза 2: Слой данных Фронтенда (Server State)

**Цель:** Реализовать "бесконечный" запрос данных на фронтенде без виртуализации, подготовить почву для интеграции с UI.

* **Зависимости:** Завершенная Фаза 1, React (Vite), TanStack Query.
* **Вход:** Работающий эндпоинт `/posts`.
* **Выход/Артефакты:**
  * React-приложение с настроенным `QueryClient`.
  * Кастомный хук `useInfinitePosts` на базе `useInfiniteQuery`.
  * Простой список (без виртуализации), выводящий `items` из всех загруженных страниц.
* **Definition of Done (DoD):**
  * [ ] Данные подгружаются "пачками" по 20 штук при нажатии на кнопку "Load More".
  * [ ] Ввод текста в Search Input инициирует новый запрос, сбрасывая текущий список и курсор.
  * [ ] Типизация (TypeScript) полностью синхронизирована между бэкендом и фронтендом.

## Фаза 3: Ядро Виртуализации (Core Virtualization)

**Цель:** Интеграция `TanStack Virtual` для управления отображением элементов в зависимости от позиции скролла.

* **Зависимости:** Фаза 2, библиотека `@tanstack/react-virtual`.
* **Вход:** Массив данных из `useInfiniteQuery`.
* **Выход/Артефакты:**
  * Компонент `VirtualFeed`.
  * Стилизация контейнера-скролла и абсолютное позиционирование элементов.
* **Definition of Done (DoD):**
  * [ ] Количество DOM-узлов карточек в инспекторе браузера остается константным (например, 10-15 штук) при любом количестве загруженных данных.
  * [ ] Скроллбар отображает корректный прогресс относительно общего (предполагаемого) объема данных.
  * [ ] При быстром скролле отсутствуют критические "белые пятна" (настроен `overscan`).

## Фаза 4: Динамические размеры и Real-World оптимизация

**Цель:** Самый сложный этап — корректная работа со "прыгающим" контентом и динамическим изменением высоты.

* **Зависимости:** Фаза 3.
* **Вход:** Данные с переменным контентом и `aspectRatio`.
* **Выход/Артефакты:**
  * Реализация `measureElement` (автоматический замер высоты DOM-узла).
  * Компонент `PostCard` с использованием CSS `aspect-ratio`.
  * Кнопка "Expand" внутри карточки, меняющая высоту элемента "на лету".
* **Definition of Done (DoD):**
  * [ ] При загрузке изображений список не меняет позицию скролла (отсутствие Layout Shift).
  * [ ] После нажатия "Expand" и изменения высоты карточки, виртуализатор мгновенно пересчитывает позиции всех элементов ниже.
  * [ ] Приложение выдерживает быстрый скролл в течение 2-х минут без утечек памяти и падения FPS ниже 50.
